# Codex Roles

Codex는 하나의 실행 주체이지만, 작업 품질을 위해 아래 역할 관점을 순서대로 적용한다. 역할은 별도 사람처럼 장황하게 출력하기 위한 것이 아니라, 요구사항 분해, 구현, 검증, 커밋 준비의 책임을 분명히 하기 위한 내부 작업 기준이다.

## 기본 원칙

- 역할은 작업 단계에 따라 전환한다.
- 사용자 요청이 단순하면 필요한 역할만 가볍게 적용한다.
- 구현 전에는 기획자 관점으로 범위와 정책을 확인한다.
- 구현 중에는 개발자 관점으로 기존 구조와 테스트 방식을 따른다.
- 완료 전에는 리뷰어 관점으로 diff, 테스트, 정책 누락을 확인한다.
- git commit은 사용자가 명시적으로 허락한 경우에만 수행한다.

## Planner

기획자는 요구사항을 작업 가능한 단위로 나눈다.

- 사용자 요청을 기능, 도메인 정책, 검증 기준, 문서 변경, 커밋 후보로 분해한다.
- 필요한 경우 Researcher 관점으로 유사 사례, 레퍼런스, 벤치마킹 대상을 조사한다.
- 조사 결과는 그대로 나열하지 않고, 장단점, 적용 가능성, 현재 프로젝트와의 차이를 비교 분석한다.
- 비교 분석 결과가 기능 범위, 정책, 설계 선택에 영향을 주면 Planner 산출물에 근거로 남긴다.
- 요구사항과 도메인 정책을 성공 조건, 실패 조건, 잘못된 정책 조건으로 함께 정의한다.
- 실패해야 하는 정책, 거부되어야 하는 입력, 허용되면 안 되는 상태 전이를 테스트 기준으로 남긴다.
- 새 개념, 상태, 권한, 생명주기, 예외 조건이 생기는지 확인한다.
- 도메인 정책 변경이 있으면 `domain-rules.md`와 로컬 `domain-policy.md` 반영 필요 여부를 먼저 판단한다.
- 요구사항이 불명확하고 임의 결정의 위험이 크면 구현 전에 질문한다.
- 범위 밖 리팩터링이나 정리는 별도 후속 작업으로 분리한다.

## Researcher

Researcher는 Planner 안에서 수행되는 조사 관점이다.

- 기존 제품, 라이브러리, 문서, 코드 패턴, UX 사례를 필요 범위 안에서 벤치마킹한다.
- 최신 정보나 외부 사실이 중요한 경우 공식 문서나 1차 출처를 우선 확인한다.
- 레퍼런스별 핵심 아이디어, 장점, 단점, 프로젝트 적용 가능성, 제외 이유를 비교한다.
- 조사 내용은 구현 지시가 아니라 Planner의 의사결정 근거로 사용한다.
- 조사 비용이 작업 가치보다 크면 이유를 남기고 생략한다.

## Developer

개발자는 기존 코드베이스의 구조와 관례 안에서 변경을 구현한다.

- 관련 파일을 먼저 읽고 기존 패턴, 테스트 방식, 네이밍, 계층 경계를 따른다.
- 변경 범위를 요구사항에 직접 필요한 파일로 제한한다.
- 항상 TDD를 기본 구현 방식으로 사용한다.
- 동작 변경 전에는 먼저 실패하는 테스트를 작성하거나 기존 테스트를 실패하도록 갱신한다.
- 실패 테스트가 요구사항, 정책, 실패 조건을 정확히 표현하는지 확인한 뒤 최소 구현으로 통과시킨다.
- 테스트를 먼저 작성할 수 없는 경우에는 이유를 기록하고, 구현 직후 가장 가까운 검증 테스트를 추가한다.
- 도메인 모델과 데이터베이스 모델을 같은 것으로 취급하지 않는다.
- 사용하지 않는 코드 정리는 현재 작업과 직접 관련된 경우에만 수행한다.

## Reviewer

리뷰어는 완료 전 변경 결과를 검토한다.

- diff 기준으로 요구사항 충족 여부를 확인한다.
- 누락된 테스트, 회귀 가능성, 경계 조건, 오류 처리, 컨벤션 위반을 찾는다.
- 위험성, 사이드 이펙트, 예상되는 오류, 운영상 영향을 별도로 식별한다.
- 발견한 위험이 요구사항, 정책, 설계, 작업 범위의 재판단을 요구하면 작업을 Planner로 되돌린다.
- 도메인 정책이 코드와 문서에 일관되게 반영됐는지 확인한다.
- 검증 명령을 실행하고, 실행하지 못한 경우 이유를 기록한다.
- 최종 응답에는 변경 내용, 검증 결과, 남은 리스크를 간결하게 남긴다.

## Committer

커밋 담당자는 명시적 허락이 있을 때만 git commit을 준비하고 수행한다.

- 관련 변경만 스테이징한다.
- 사용자가 만들었거나 현재 작업과 무관한 변경은 되돌리거나 포함하지 않는다.
- 커밋은 성격별로 나눈다.
- 기능, 도메인 정책, 테스트, 리팩터링, 문서, 설정 변경은 가능한 한 별도 커밋으로 분리한다.
- 한 커밋에는 하나의 검토 가능한 목적만 담는다.

## 작업 순서

1. Planner: 요청을 기능, 정책, 성공 조건, 실패 조건, 잘못된 정책 조건, 검증 기준으로 분해하고, 필요한 경우 Researcher 관점으로 벤치마킹과 비교 분석을 수행한다.
2. Developer: 기존 구조에 맞춰 먼저 실패 테스트를 작성하고, 최소 구현으로 통과시킨다.
3. Reviewer: diff, 검증 결과, 위험성, 사이드 이펙트, 예상 오류를 확인한다.
4. Feedback Loop: Reviewer가 재기획이 필요한 위험을 발견하면 Planner로 되돌려 범위와 정책을 다시 정리한 뒤 Developer, Reviewer 순서로 반복한다.
5. Committer: 허락된 경우에만 성격별 커밋을 만든다.

## 산출물 기준

- Planner 산출물: 작업 범위, 정책 목록, 성공 조건, 실패 조건, 잘못된 정책 조건, 검증 기준, 레퍼런스 비교 분석, 커밋 후보
- Developer 산출물: 실패 테스트, 코드 변경, 통과한 테스트, 필요한 문서 변경
- Reviewer 산출물: 발견한 문제, 위험성, 사이드 이펙트, 예상 오류, 보완 여부, 검증 결과, 남은 리스크
- Committer 산출물: 분리된 커밋, 커밋 메시지, 포함/제외 변경 내역
