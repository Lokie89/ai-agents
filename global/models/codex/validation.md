# 검증 방법

## 기본 체크리스트

- [ ] 청사진 요청인 경우 실제 작업을 실행하지 않았다.
- [ ] 청사진에 작업 목적, 접근 방식, 변경점, 사이드이펙트, 검증 방법이 포함되었다.
- [ ] 개념 추가가 포함된 경우 도메인 정책을 먼저 확인했다.
- [ ] 도메인 정의가 불명확한 경우 구현 전에 질의를 제시했다.
- [ ] 도메인 모델과 데이터베이스 모델을 같은 것으로 취급하지 않았다.
- [ ] 역할별 모델 선택이 `model-routing.md` 기준과 일치한다.
- [ ] 요청한 목표가 실제 변경에 반영되었다.
- [ ] 관련 없는 파일 변경이 없다.
- [ ] 작업 범위 안의 사용하지 않는 코드, 죽은 분기, 불필요한 주석을 정리했다.
- [ ] 기존 스타일과 패턴을 따랐다.
- [ ] 오류 처리, 경계 조건, 실패 경로를 고려했다.
- [ ] 문서가 필요한 경우 함께 갱신했다.
- [ ] 테스트 또는 빌드 명령을 실행했다.
- [ ] 실행하지 못한 검증은 이유를 기록했다.

## 코드 변경 검증

프로젝트에 해당 명령이 있으면 아래 순서로 실행한다.

1. 포맷 또는 린트
2. 타입 체크
3. 단위 테스트
4. 통합 테스트
5. 빌드

## 모델 라우팅 검증

- 역할별 모델 선택이 `global/models/codex/model-routing.md`의 루브릭과 일치하는지 확인한다.
- 모델 지정이 별칭, 풀 모델 ID, `inherit` 중 하나인지 확인한다.
- 모델을 생략한 경우 `inherit`로 해석되는지 확인한다.
- 에이전트 frontmatter에 `type` 필드가 없는지 확인한다.
- 실제 모델 ID가 필요한 경우 계정, 표면, 시점에 따른 가용성을 확인한다.

## 도메인 정책 검증

- 도메인 관련 변경이 `global/models/codex/domain-rules.md`를 참조했는지 확인한다.
- 실제 도메인 정책 변경이 필요했다면 `local/<project-name>/domain-policy.md`가 함께 갱신되었는지 확인한다.
- 새로 추가한 개념의 이름과 의미가 도메인 정책과 일치하는지 확인한다.
- 엔티티, 상태, 권한, 생명주기, 불변 조건이 문서나 코드에 일관되게 반영되었는지 확인한다.
- 도메인 정책이 불명확했던 부분은 사용자 질의와 결정 내용을 기록한다.
- 기술적 편의 때문에 도메인 의미가 왜곡되지 않았는지 확인한다.

## 데이터베이스 모델링 검증

- 도메인 모델이 개발자가 사용하는 개념과 정책을 표현하는지 확인한다.
- 데이터베이스 모델이 DBMS의 저장, 조회, 인덱싱, 제약 관리에 적합한지 확인한다.
- 테이블 구조를 이유로 도메인 개념이 왜곡되지 않았는지 확인한다.
- 도메인 모델과 데이터베이스 모델이 다르면 매핑 이유와 트레이드오프를 기록한다.
- 마이그레이션, 인덱스, 제약 조건 변경의 사이드이펙트를 확인한다.

## 보이스카웃 규칙 검증

- 정리한 코드가 현재 작업 범위와 직접 관련되는지 확인한다.
- 삭제한 코드가 외부 API, 설정, 런타임 참조로 사용되지 않는지 확인한다.
- 죽은 코드 제거가 동작 변경을 만들 수 있다면 테스트나 근거를 남긴다.
- 범위 밖 정리가 필요해 보이면 이번 변경에 섞지 않고 남은 작업으로 기록한다.

## 문서 변경 검증

- 링크가 올바른 상대 경로를 가리키는지 확인한다.
- 문서 구조가 모델 기준 5가지를 모두 포함하는지 확인한다.
- 다음 세션이 읽어도 작업 목적과 남은 일이 명확한지 확인한다.

## 청사진 검증

- 실행 명령, 파일 수정, 외부 호출이 발생하지 않았는지 확인한다.
- 작업 내용이 사용자의 명령과 직접 연결되는지 확인한다.
- 변경되는 파일이나 동작이 구체적으로 설명되었는지 확인한다.
- 사이드이펙트가 기능, 데이터, 성능, 보안, 운영 관점에서 검토되었는지 확인한다.
- 사용자가 실행 여부를 판단할 수 있을 만큼 검증 방법과 완료 기준이 명확한지 확인한다.

## 결과 보고 형식

- 변경한 파일
- 핵심 변경 내용
- 확인한 도메인 정책 또는 남은 도메인 질문
- 수정한 로컬 도메인 정책 파일
- 도메인 모델과 데이터베이스 모델의 분리 여부
- 적용한 모델 선택 기준
- 실행한 검증 명령과 결과
- 실행하지 못한 검증과 이유
- 정리한 사용하지 않는 코드 또는 정리하지 않은 이유
- 남은 작업
