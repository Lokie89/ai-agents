# 도메인 규칙

이 문서는 Codex 모델이 실제 프로젝트 도메인 정책을 어떻게 확인하고 수정해야 하는지 정의하는 글로벌 규칙입니다.

실제 도메인 정책은 `local/<project-name>/domain-policy.md`에 둡니다.

## 기본 원칙

- 모든 개념 추가와 모델링 작업에서는 도메인 정책을 가장 우선한다.
- 도메인의 정의, 용어, 경계, 정책이 뚜렷하지 않으면 구현보다 질의를 먼저 수행한다.
- 도메인 관련 판단 방식은 이 파일을 참조한다.
- 실제 도메인 정책 자체가 변경되어야 하면 로컬 `domain-policy.md`를 먼저 갱신하고, 코드와 문서는 갱신된 정책을 따른다.

## 도메인 정의 기준

- 도메인은 개발자가 사용하는 업무 개념을 구체화한 모델이다.
- 도메인은 용어, 규칙, 상태, 권한, 생명주기, 불변 조건을 포함할 수 있다.
- 새 개념은 기존 도메인 용어와 충돌하지 않아야 한다.
- 정의가 모호하면 선택지를 임의로 고정하지 않고 사용자에게 질문한다.

## 도메인 모델과 데이터베이스 모델

- 도메인 모델은 업무 개념, 규칙, 정책, 생명주기를 표현한다.
- 데이터베이스 모델은 DBMS에서 저장, 조회, 인덱싱, 무결성 제약을 효율적으로 다루기 위한 구조다.
- 데이터베이스 테이블 구조를 도메인 개념의 정답처럼 취급하지 않는다.
- 도메인 개념을 먼저 정의한 뒤, 필요한 경우 DBMS 특성에 맞게 테이블, 컬럼, 인덱스, 제약 조건으로 변환한다.
- 도메인 모델과 데이터베이스 모델이 다르면 매핑 이유와 트레이드오프를 기록한다.

## 질의가 필요한 경우

- 새 용어의 의미가 기존 용어와 겹치거나 충돌한다.
- 엔티티, 값 객체, 상태, 정책 중 무엇으로 볼지 불명확하다.
- 생명주기나 상태 전이가 명확하지 않다.
- 권한, 소유권, 책임 경계가 불명확하다.
- DBMS 편의를 위해 도메인 의미를 바꿔야 하는지 판단이 필요하다.
