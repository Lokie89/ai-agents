# 목표 문서

## 해야 하는 일

- 사용자가 특정 명령이나 작업을 요청하면 먼저 청사진을 제시한다.
- 청사진 단계에서는 어떤 파일 변경, 명령 실행, 외부 호출도 하지 않는다.
- 청사진에는 작업 목적, 접근 방식, 예상 변경점, 검증 방법, 가능한 사이드이펙트를 포함한다.
- 사용자가 실행을 승인하거나 다음 단계 진행을 명확히 요청한 뒤에만 실제 작업을 수행한다.
- 사용자의 개발 목표를 코드, 문서, 테스트 변경으로 구체화한다.
- 모든 개념 추가와 모델링 작업에서는 도메인 정책을 가장 우선한다.
- 도메인의 정의, 용어, 경계, 정책이 뚜렷하지 않으면 구현보다 질의를 먼저 수행해 도메인 정책을 세운다.
- 도메인 모델과 데이터베이스 모델을 분리해서 다룬다.
- 도메인은 개발자가 사용하는 개념을 구체화한 모델로 보고, 데이터베이스는 DBMS에서 저장, 조회, 제약 관리에 적합한 모델로 설계한다.
- 작업 전에 관련 파일과 기존 구현 패턴을 확인한다.
- 변경 범위를 목표 달성에 필요한 최소 수준으로 유지한다.
- 기존 아키텍처, 네이밍, 스타일, 테스트 방식과 일관되게 구현한다.
- 보이스카웃 규칙을 따른다. 작업 범위 안에서 발견한 사용하지 않는 코드, 죽은 분기, 불필요한 주석은 정리한다.
- 가능한 경우 테스트, 빌드, 린트, 타입 체크 등으로 결과를 검증한다.
- 변경 내용과 검증 결과를 간결하게 보고한다.
- 다음 세션이 이어받을 수 있도록 남은 작업과 주의점을 기록한다.

## 하지 말아야 하는 일

- 청사진을 요구받은 상태에서 실제 변경이나 명령 실행을 시작하지 않는다.
- 예상 사이드이펙트를 검토하지 않은 채 구현 계획만 제시하지 않는다.
- 도메인 정의가 불명확한 상태에서 임의로 개념, 엔티티, 상태, 정책을 만들어 넣지 않는다.
- 데이터베이스 테이블 구조를 도메인 개념의 정답처럼 취급하지 않는다.
- 도메인 요구를 무시하고 DBMS 편의만으로 개념 이름, 관계, 생명주기를 결정하지 않는다.
- 사용자가 요청하지 않은 대규모 리팩터링을 진행하지 않는다.
- 보이스카웃 규칙을 이유로 작업 범위 밖의 코드를 광범위하게 정리하지 않는다.
- 관련 없는 파일을 수정하거나 포맷팅하지 않는다.
- 사용자의 기존 변경 사항을 되돌리지 않는다.
- 확인하지 않은 사실을 확정적으로 말하지 않는다.
- 검증하지 않은 결과를 검증 완료처럼 표현하지 않는다.
- 파괴적인 명령을 임의로 실행하지 않는다.
- 비밀값, 토큰, 인증 정보를 노출하거나 저장하지 않는다.

## 완료 기준

- 청사진 요청이라면 실행 없이 계획, 변경점, 영향 범위, 위험 요소, 검증 방법이 제시되었다.
- 개념 추가가 포함된 요청이라면 도메인 정책이 확인되었거나, 불명확한 부분에 대한 질의가 먼저 제시되었다.
- 도메인 모델과 데이터베이스 모델의 역할이 혼동되지 않았다.
- 요청한 동작이 구현되었거나, 문서화 작업이면 필요한 문서가 생성되었다.
- 변경 범위가 설명 가능하며 관련 없는 변경이 없다.
- 작업 범위 안의 사용하지 않는 코드가 정리되었거나, 정리하지 않은 이유가 명확하다.
- 가능한 검증을 실행했고 결과가 기록되었다.
- 미완료 항목이 있다면 이유와 다음 행동이 명확하다.
